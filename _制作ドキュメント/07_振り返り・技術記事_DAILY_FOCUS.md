## 1. プロジェクト振り返り

### 企画・設計段階の振り返り

マーケットリサーチ、ペルソナ設計、サイトマップ作成を通して、サービスを作る前に「誰のどんな課題を解決するのか」を言語化する重要性を学んだ。特に DAILY FOCUS では、タスク管理アプリ全体ではなく「今日やることを一つに絞る」という価値に焦点を当てたことで、サービスの軸が明確になった。一方で、市場規模や競合分析は情報の切り分けが難しく、どこまでを根拠として扱うかに苦労した。次回は最初に比較軸と調査対象を明確に決めてから調査を進め、設計の精度をさらに高めたい。

### デザイン段階の振り返り

デザインガイドラインとワイヤーフレームを作成する中で、見た目の良さだけでなく、情報の優先順位やユーザーの視線の流れを意識することの大切さを実感した。トップページでは、ヒーロー、機能紹介、継続しやすさ、CTA の順に情報を並べることで、サービス理解から登録までの流れを自然に作れたと感じている。一方で、Figma の操作やテンプレートの扱いには時間がかかり、思ったよりも手を動かす量が多かった。次回は最初から必要な画面数を絞り、テンプレート活用と構成設計を先に固めてから作業したい。

### フロントエンド実装の振り返り

フロントエンドでは、画面の見た目だけでなく、ユーザーが迷わず操作できる流れを意識して設計することが重要だと学んだ。特にログイン、新規登録、タスク一覧、編集画面などは、入力項目やボタンの位置を整理するだけで使いやすさが大きく変わることを実感した。また、ワイヤーフレームから実装イメージを考える流れを経験できたことは大きな収穫だった。一方で、画面ごとの役割整理やコンポーネントの分け方はまだ甘い部分があるため、今後は HTML/CSS/JavaScript だけでなく、再利用性を意識した実装力も高めたい。

### バックエンド実装の振り返り

今回は企画・設計・画面設計の比重が大きく、Java / Spring Boot の本格実装までは進めていないが、ログイン、タスク管理、振り返り記録といった機能を支えるバックエンドの役割について具体的に考える機会になった。特に、認証、CRUD、データ保存、集計処理などは、画面設計と同じくらい整理が重要だと感じた。今後 Spring Boot で実装する際は、画面ごとに必要な API やデータ構造を先に設計し、フロントエンドと整合性を取りながら開発を進めたい。

### 全体を通しての学び

このプロジェクトを通して最も学んだことは、「作り始める前の整理」が完成度を大きく左右するということである。アイデアだけではなく、マーケットリサーチ、ペルソナ、サイトマップ、ワイヤーフレーム、デザインガイドラインを順に作ることで、サービスの方向性を少しずつ具体化できた。また、AIを使うことで文章の整理や設計の補助は大きく進んだが、最終的に何を採用するか判断するのは自分自身だと実感した。特に今回は Figma 作業や画面設計で苦労したが、その分「ユーザーにどう見えるか」を意識できたことが大きな成長につながった。今後は設計から実装までをより一貫してつなげられるようにしたい。

## 2. 技術記事

### 技術記事タイトル

Figmaでワイヤーフレームを最短で形にするために意識したこと

### 記事本文

#### 1. はじめに

今回の DAILY FOCUS 制作では、トップページやログイン画面などのワイヤーフレームを Figma で作成した。最初は「とにかく見た目を整えなければいけない」と考えていたが、実際に取り組んでみると、重要なのは完成度よりも「画面の構成が伝わること」だと分かった。本記事では、Figma に慣れていない状態から、どのように最短でワイヤーフレームを形にしていったかをまとめる。

#### 2. 課題

苦労した点は主に3つあった。  
1つ目は、Figma の操作に慣れておらず、画面を一から組むだけでも時間がかかったこと。  
2つ目は、どこまで作り込めば「提出物として十分」なのか判断しづらかったこと。  
3つ目は、トップページ・下層ページ・ログイン画面など複数の画面を限られた時間で作る必要があったことだ。

#### 3. 調査

最初はゼロから作ろうとしたが、時間がかかりすぎると感じたため、既存のテンプレートや参考画面を見ながら構造を流用する方針に切り替えた。特に参考になったのは、LP系の基本構成と、ログイン画面の中央カード型レイアウトである。そこから、必要な要素だけを DAILY FOCUS 用に置き換える形にしたことで、作業効率が大きく上がった。

#### 4. 実装

ワイヤーフレームでは、まず「必要なセクションを決める」ことを優先した。トップページでは、以下のような構成に整理した。

```html
<main>
  <header>ロゴ / ナビゲーション / CTA</header>
  <section id="hero">キャッチコピー / 説明 / メインCTA</section>
  <section id="features">機能紹介</section>
  <section id="reason">続けやすい理由</section>
  <section id="cta">登録促進</section>
  <footer>規約 / サポート / お問い合わせ</footer>
</main>
```

このように HTML の構造として考えると、Figma 上でも「どの箱をどこに置けばいいか」が分かりやすくなった。また、ログイン画面や新規登録画面も、中央にカードを置き、その中に入力欄とボタンを縦に並べるだけで十分形になった。

```html
<section class="auth-card">
  <h1>ログイン</h1>
  <input type="email" placeholder="メールアドレス" />
  <input type="password" placeholder="パスワード" />
  <button>ログインする</button>
</section>
```

コードとして考えることで、Figma でも「見た目を作る」のではなく「構造を作る」意識を持てた。

#### 5. 結果

結果として、トップページ、ログイン、新規登録、一覧、詳細、編集といった必要な画面を、最低限の工数でそろえることができた。すべてを完璧に作り込むのではなく、レイアウトの流れと要素の役割が分かるレベルを目指したことで、提出物として十分な形にまとめられた。

#### 6. 学び

今回の経験から学んだのは、Figma では「完成デザイン」をいきなり目指すより、「情報設計を視覚化する」ことを優先した方が進めやすいということだ。また、テンプレートや参考画面をうまく使い、必要な部分だけを自分のサービスに合わせて置き換えることは、制作効率を上げるうえで非常に有効だった。今後は、Figma の操作そのものにも慣れつつ、画面設計と実装をつなげて考える力をさらに高めていきたい。

## 4. 今後の学習計画

### 今後学びたい技術・分野

#### 短期的に学ぶべきこと（3ヶ月以内）

- Spring Boot を使ったログイン認証と CRUD 実装
- HTML/CSS のレイアウト設計とコンポーネント整理

- UI/UX の改善を意識した画面設計
- ポートフォリオ向けに、設計から実装まで完了したWebアプリ制作

- Spring Boot 公式ドキュメント
- MDN Web Docs
- Progate / ドットインストール / Udemy などの基礎講座
